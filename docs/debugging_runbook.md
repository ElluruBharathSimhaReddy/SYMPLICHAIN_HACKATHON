# Monday 9 AM POD outage runbook

## Goal

Find where the failure happens in this chain:

Driver app -> Django API -> S3 -> Celery -> Bedrock -> RDS

## Step 1: Confirm the issue at the API entry point

### Why first

If Django never received the request, there is no reason to inspect Celery, Bedrock, or RDS yet.

### What to check

- Nginx access logs for incoming upload requests
- Django or gunicorn error logs for request failures
- HTTP status codes, request latency, and payload size issues

### Useful commands

```bash
sudo tail -f /var/log/nginx/access.log
sudo tail -f /var/log/nginx/error.log
sudo journalctl -u gunicorn -n 200 --no-pager
sudo journalctl -u gunicorn -f
```

### Expected findings

- 4xx suggests client, auth, payload, or validation issue
- 5xx suggests server-side failure before or during upload handling

## Step 2: Check whether files reached S3

### Why second

POD upload failures often come from bad AWS credentials, bucket policy changes, expired IAM permissions, or region mismatch.

### What to check

- Django log lines around `put_object` or storage backend calls
- CloudWatch logs or metrics for application exceptions
- S3 bucket object count for recent uploads

### Useful checks

- Inspect CloudWatch log stream for stack traces mentioning boto3, credentials, timeout, access denied
- Compare one failing shipment id against S3 keys generated by the app

## Step 3: Verify Celery task creation and worker health

### Why third

If S3 succeeded but validation never happened, the break is around task enqueue or task execution.

### What to check

- Flower dashboard for queued, active, retried, and failed tasks
- Redis broker availability
- Celery worker logs

### Useful commands

```bash
sudo journalctl -u celery -n 200 --no-pager
sudo journalctl -u celery -f
celery -A config inspect active
celery -A config inspect reserved
celery -A config inspect stats
```

### Questions to answer

- Are tasks being created at all?
- Are tasks stuck in queued state?
- Are workers alive and consuming?
- Are there repeated task retries?

## Step 4: Inspect Bedrock call failures

### Why fourth

If Celery is running but validation tasks fail during processing, Bedrock timeout or auth issues become likely.

### What to check

- CloudWatch logs for task stack traces
- AWS request errors such as throttling, access denied, model not found, timeout
- Recent IAM or secret changes

### Typical symptoms

- spike in task retry count
- long task duration
- explicit boto3 or Bedrock client exceptions

## Step 5: Check database write stage in RDS

### Why fifth

If inference succeeds but final status is not saved, customers may still see the upload as failed or pending forever.

### What to check

- DB connection saturation
- lock contention
- failed insert or update statements
- RDS CPU, connections, storage, and slow query dashboards

### Useful places

- RDS Performance Insights
- RDS metrics in CloudWatch
- Django logs showing OperationalError or transaction rollback

## Decision tree

- No request in Nginx: mobile client, DNS, ALB, or network path issue
- Request reaches Django but S3 write fails: credentials, IAM, bucket policy, region, request size
- S3 works but no task: Celery enqueue or Redis issue
- Task exists but fails before completion: Bedrock dependency or task code bug
- Task succeeds but no final status: RDS write failure
